# Index (데이터베이스 인덱스)

## 1. 개념 정리
인덱스(Index)는
데이터베이스 테이블에서
데이터를 빠르게 조회하기 위해 사용하는
자료구조이다.

책의 목차처럼,
테이블의 특정 컬럼 값을 기준으로
데이터 위치를 미리 정리해 두는 역할을 한다.

일반적으로 B-Tree 구조를 사용하며,
인덱스를 통해 전체 테이블을 스캔하지 않고도
필요한 데이터를 빠르게 찾을 수 있다.

즉,
- 인덱스는 조회 성능을 높이기 위한 구조
- 대신 저장 공간과 쓰기 비용이 추가로 발생한다.

---

## 2. 왜 중요한가
백엔드 서버의 성능 문제는
대부분 DB 조회 성능에서 발생한다.

인덱스를 적절히 사용하지 않으면:
- 테이블 풀 스캔 발생
- 응답 시간 급증
- DB 부하 증가
- 전체 서비스 성능 저하

특히 데이터가 수십만, 수백만 건 이상 쌓이는 서비스에서는
인덱스 유무가 곧 서비스 안정성과 직결된다.

---

## 3. 백엔드 실무 관점
실무에서 인덱스는 다음과 같은 상황에서 핵심이 된다.

- WHERE 조건으로 자주 조회되는 컬럼
- JOIN 조건에 사용되는 컬럼
- 정렬(ORDER BY) 또는 범위 조회

Spring 기반 백엔드에서는
API 응답 속도가 느릴 경우,
로직보다 먼저 SQL 실행 계획과 인덱스 여부를 확인한다.

실무 포인트는 다음과 같다.

- 인덱스는 읽기 성능을 높인다
- INSERT / UPDATE / DELETE 시에는 오히려 비용이 증가한다
- 모든 컬럼에 인덱스를 거는 것은 오히려 성능 저하를 유발한다

---

## 4. 트레이드오프 / 주의사항
- 인덱스가 많을수록 쓰기 성능은 저하된다
- 잘못된 인덱스는 사용되지 않고 공간만 차지한다
- 조건 순서, 컬럼 선택에 따라 인덱스 활용 여부가 달라진다

즉,
> 인덱스는 많을수록 좋은 것이 아니라,
> 조회 패턴에 맞게 최소한으로 설계해야 한다.

라는 트레이드오프가 존재한다.

---

## 5. 예상 면접 질문 & 답변 연습

### Q1. 인덱스를 사용하는 이유는 무엇인가요?
A.
테이블 전체를 스캔하지 않고도
조건에 맞는 데이터를 빠르게 조회해
DB 조회 성능을 향상시키기 위해 사용합니다.

---

### Q2. 인덱스를 많이 만들면 항상 성능이 좋아지나요?
A.
아닙니다.
조회 성능은 좋아질 수 있지만,
쓰기 작업 시 인덱스 갱신 비용이 발생해
오히려 전체 성능이 저하될 수 있습니다.

---

### Q3. 인덱스가 있는데도 조회가 느릴 수 있나요?
A.
네.
조건 컬럼이 인덱스를 타지 않거나,
인덱스 순서와 다른 조건을 사용할 경우
풀 스캔이 발생할 수 있습니다.

---

## 6. 한 문장 요약 (면접용)
인덱스는 조회 성능을 높이기 위한 핵심 구조이지만,
쓰기 비용과 조회 패턴을 고려해
신중하게 설계해야 하는 트레이드오프 요소다.
