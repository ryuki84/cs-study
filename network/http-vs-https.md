# HTTP / HTTPS

## 1. 개념 정리
HTTP(HyperText Transfer Protocol)는
클라이언트와 서버가 요청(Request)과 응답(Response)을 주고받는
애플리케이션 계층 프로토콜이다.
TCP 위에서 동작하며,
기본적으로 데이터는 암호화되지 않은 평문 형태로 전송된다.

HTTPS(HyperText Transfer Protocol Secure)는
HTTP에 TLS(Transport Layer Security)를 적용한 프로토콜로,
통신 내용을 암호화하여 보안성을 강화한 방식이다.
서버 인증서를 통해 서버의 신원을 검증하고,
데이터 도청이나 위·변조를 방지한다.

즉,
- HTTP는 평문 통신
- HTTPS는 암호화된 안전한 통신

이라고 볼 수 있다.

---

## 2. 왜 중요한가
백엔드 서버는 대부분 HTTP 기반 API를 제공하며,
통신 보안은 서비스 신뢰성과 직결된다.

HTTP를 그대로 사용할 경우:
- 네트워크 구간에서 데이터 도청 가능
- 인증 정보 탈취 위험
- 중간자 공격(MITM) 가능성

이 존재한다.

특히 로그인, 결제, 개인정보를 다루는 서비스에서는
HTTPS 미적용 자체가 심각한 보안 취약점이 된다.
현재 대부분의 서비스에서 HTTPS는 선택이 아닌 필수다.

---

## 3. 백엔드 실무 관점
실무에서 HTTPS를 적용하면 다음과 같은 변화가 발생한다.

- 클라이언트와 서버 간 통신이 TLS로 암호화됨
- 서버는 인증서를 통해 자신의 신원을 증명
- 초기 연결 시 TLS 핸드셰이크 비용 발생

하지만 한 번 연결이 성립된 이후에는
Keep-Alive와 세션 재사용 등을 통해
성능 비용을 최소화할 수 있다.

실제 운영 환경에서는:
- 모든 외부 API 통신은 HTTPS 사용
- HTTP 요청은 HTTPS로 리다이렉트
- 인증, 결제, 개인정보 API는 HTTPS만 허용

과 같은 정책을 적용한다.

---

## 4. 트레이드오프 / 주의사항
- HTTPS는 보안성이 높지만,
  초기 TLS 핸드셰이크로 인한 성능 비용이 발생한다.
- 인증서 관리(갱신, 만료) 이슈가 존재한다.

그러나 현대 서버 환경에서는:
- 하드웨어 성능 향상
- Keep-Alive 및 TLS 세션 재사용

으로 인해 HTTPS 성능 비용은 대부분 무시 가능한 수준이다.

즉,
> 보안을 위해 HTTPS를 사용하되,
> 성능 문제는 설계로 해결하는 것이 일반적인 접근이다.

---

## 5. 예상 면접 질문 & 답변 연습

### Q1. HTTP와 HTTPS의 가장 큰 차이는 무엇인가요?
A.
HTTP는 평문으로 데이터를 전송하는 반면,
HTTPS는 TLS를 적용해 통신 내용을 암호화하고
서버 인증을 통해 보안성을 강화합니다.

---

### Q2. HTTPS를 사용하면 무조건 느려지지 않나요?
A.
초기 연결 시 TLS 핸드셰이크 비용은 존재하지만,
Keep-Alive와 세션 재사용을 통해
실제 서비스에서는 성능 영향이 크지 않습니다.

---

### Q3. HTTP를 아예 사용하면 안 되는 이유는 무엇인가요?
A.
HTTP는 네트워크 구간에서 데이터가 그대로 노출되기 때문에,
인증 정보 탈취나 중간자 공격 위험이 있어
보안이 필요한 서비스에는 적합하지 않습니다.

---

## 6. 한 문장 요약 (면접용)
HTTP는 평문 기반 통신 프로토콜이고,
HTTPS는 TLS를 적용해 **보안과 신뢰성을 확보한 통신 방식**이다.
