# Blocking / Non-Blocking, Sync / Async

## 1. 개념 정리

### Blocking / Non-Blocking
Blocking과 Non-Blocking은 **호출한 작업이 제어권을 반환하는 시점**의 차이다.

- Blocking: 작업이 끝날 때까지 호출한 스레드가 대기
- Non-Blocking: 작업이 끝나지 않아도 즉시 제어권 반환

---

### Sync / Async
Sync와 Async는 **작업 완료를 누가 책임지고 확인하느냐**의 차이다.

- Sync: 호출한 쪽이 작업 완료를 직접 확인
- Async: 작업 완료를 다른 주체(콜백, 이벤트 등)가 처리

---

## 2. 왜 중요한가
백엔드 서버에서 가장 중요한 자원은 **스레드**다.  
Blocking 구조에서는 I/O 대기 동안 스레드가 아무 일도 하지 못한 채 점유된다.

이로 인해:
- 스레드 고갈
- 응답 지연
- 장애로 확산

이 발생할 수 있다.

---

## 3. 백엔드 실무 관점
전통적인 Spring MVC는 기본적으로 **Blocking I/O** 구조다.

- DB 조회
- 외부 API 호출
- 파일 I/O

동안 스레드가 대기한다.

트래픽이 많아질수록:
- 스레드 수 증가
- 컨텍스트 스위칭 증가
- 성능 저하

로 이어질 수 있다.

---

## 4. 트레이드오프 / 주의사항
- Non-Blocking/Async는 복잡도가 증가한다
- 디버깅과 흐름 추적이 어렵다
- 모든 서비스에 적합하지는 않다

즉,
> **단순한 서비스는 Blocking,  
고부하 서비스는 Non-Blocking 고려**

가 일반적인 선택이다.

---

## 5. 예상 면접 질문 & 답변 연습

### Q1. Blocking과 Non-Blocking의 차이는 무엇인가요?
A.  
Blocking은 작업 완료 전까지 호출한 스레드가 대기하고,  
Non-Blocking은 작업 완료 여부와 상관없이 즉시 제어권을 반환합니다.

---

### Q2. Sync와 Async의 차이는 무엇인가요?
A.  
Sync는 호출한 쪽이 작업 완료를 직접 관리하고,  
Async는 작업 완료 처리를 다른 주체가 담당합니다.

---

### Q3. Blocking I/O가 왜 문제가 될 수 있나요?
A.  
I/O 대기 동안 스레드가 점유되어 스레드 자원이 낭비되고,  
동시 요청이 많아지면 성능 저하나 장애로 이어질 수 있기 때문입니다.

---

## 6. 한 문장 요약 (면접용)
Blocking/Non-Blocking은 제어권 반환 시점의 차이이고,  
Sync/Async는 작업 완료 책임의 차이이며,  
백엔드 서버에서는 **스레드 효율성** 관점에서 매우 중요합니다.
